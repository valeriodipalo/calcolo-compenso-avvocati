# Topic Cluster Implementation Plan — AvvocatoTools.it

**Status**: Approved
**Date**: 2026-04-05
**Based on**: Manus SEO Analysis + current site audit

---

## Principi guida

1. **Zero URL migration** — gli URL esistenti restano flat alla root. I cluster si costruiscono tramite internal linking, breadcrumb e navigazione, non tramite gerarchia URL.
2. **Registry-driven** — tutto (navbar, homepage, related tools, breadcrumb, sitemap) si genera dal `toolRegistry.ts`. Un solo posto da aggiornare.
3. **Incrementale** — ogni fase è deployabile indipendentemente e porta valore da sola.

---

## Fase 1: Cluster Infrastructure (fondamenta)

### 1.1 Definizione dei cluster

5 cluster principali. Ogni cluster ha: `id`, `label`, `slug` (per la futura hub page), `description`, `icon`.

| Cluster ID | Label | Slug (hub page) | Tool assegnati |
|---|---|---|---|
| `compensi` | Compensi e Fatturazione | `compensi-forensi` | `calcolo-compenso`, `calcolo-fattura-avvocato` |
| `termini` | Termini e Scadenze | `termini-processuali` | `calcolo-termini-processuali`, `calcolo-termini-impugnazione` |
| `interessi` | Interessi e Rivalutazione | `interessi-e-rivalutazione` | `calcolo-interessi-legali`, `calcolo-interessi-moratori`, `calcolo-rivalutazione-monetaria` |
| `danno` | Danno e Risarcimento | `danno-e-risarcimento` | `calcolo-danno-biologico`, `calcolo-danno-non-patrimoniale` |
| `fiscale` | Fiscale e Tributario | `fiscale-e-tributario` | `contributo-unificato` |

### 1.2 Modifiche a `toolRegistry.ts`

Aggiungere una struttura `CLUSTERS` e un campo `cluster` a ogni `ToolEntry`.

```typescript
// Nuova interfaccia
export interface ToolCluster {
  id: string;
  label: string;            // "Compensi e Fatturazione"
  slug: string;             // "compensi-forensi" (URL della futura hub page)
  description: string;      // Breve descrizione del cluster
  icon: LucideIcon;
}

// Aggiunta a ToolEntry
export interface ToolEntry {
  // ... campi esistenti invariati ...
  cluster: string;          // ID del cluster (es. "compensi")
  relatedSlugs?: string[];  // Tool correlati cross-cluster (opzionale, max 2)
}
```

**`CLUSTERS` array** — definito nello stesso file, esportato. Ordine = ordine di visualizzazione in navbar e homepage.

**`relatedSlugs`** — link cross-cluster suggeriti manualmente. Esempi:
- `calcolo-danno-biologico` → `["calcolo-interessi-legali"]` (dopo il danno calcoli gli interessi)
- `calcolo-compenso` → `["calcolo-fattura-avvocato"]` (già nello stesso cluster, ma esplicita il flusso)
- `calcolo-rivalutazione-monetaria` → `["calcolo-danno-biologico"]` (rivalutazione usata spesso nei risarcimenti)

### 1.3 Helper functions

Aggiungere a `toolRegistry.ts` (o in un file separato `src/lib/clusters.ts`):

```typescript
// Tutti i tool di un cluster
function getToolsByCluster(clusterId: string): ToolEntry[]

// Il cluster di un tool
function getClusterForTool(slug: string): ToolCluster | undefined

// Tool correlati (stesso cluster + cross-cluster), escludendo il tool corrente
function getRelatedTools(slug: string): { sameCluster: ToolEntry[]; crossCluster: ToolEntry[] }
```

## Fase 2: Navigazione UX

### 2.1 Navbar raggruppata per cluster

**Desktop**: sostituire la lista piatta con dropdown per cluster.

```
[Compensi ▾] [Termini ▾] [Interessi ▾] [Danni ▾] [Fiscale ▾]
```

Ogni dropdown mostra:
- Titolo del cluster (non cliccabile, oppure link alla hub page quando esisterà)
- Lista dei tool con icona e navLabel
- Tool attivo evidenziato

Implementazione:
- Il componente itera su `CLUSTERS`, per ognuno filtra i tool con `getToolsByCluster()`
- Dropdown con hover su desktop (con `onMouseEnter`/`onMouseLeave` e un piccolo delay di chiusura ~150ms per evitare chiusure accidentali)
- Il cluster con il tool attivo ha il label evidenziato anche quando il dropdown è chiuso

**Mobile**: sostituire la lista piatta con accordion per cluster.
- Ogni cluster è una sezione collassabile
- Il cluster contenente il tool attivo parte espanso
- Tap sul titolo del cluster per espandere/collassare

**Scalabilità**: questo pattern regge fino a 40-50 tool senza problemi. Con 10 tool attuali non è strettamente necessario, ma implementarlo ora evita un refactor futuro della navbar quando i tool cresceranno.

### 2.2 Homepage raggruppata per sezioni

Sostituire la griglia piatta con sezioni per cluster:

```
[Hero esistente — invariato]

── Compensi e Fatturazione ──────────────────────
   Breve descrizione del cluster (1-2 righe).
   [Card: Calcolo Compenso]  [Card: Fattura Avvocato]

── Termini e Scadenze ──────────────────────────
   Breve descrizione del cluster (1-2 righe).
   [Card: Termini Processuali]  [Card: Termini Impugnazione]

── Interessi e Rivalutazione ───────────────────
   ...

── Danno e Risarcimento ────────────────────────
   ...

── Fiscale e Tributario ────────────────────────
   ...
```

Ogni sezione ha:
- Titolo H2 con il label del cluster
- Descrizione breve (dalla definizione del cluster)
- Griglia di card tool (layout attuale, invariato per le singole card)
- Quando esisterà la hub page: link "Scopri di più →" sotto la griglia

Le card tool restano identiche a oggi (icona, titolo, descrizione, badge, freccia).

### 2.3 Componente `RelatedTools`

Nuovo componente shared: `src/components/shared/RelatedTools.tsx`

**Props**: `currentSlug: string`

**Comportamento**:
1. Chiama `getRelatedTools(currentSlug)`
2. Mostra due sezioni:
   - "Nello stesso ambito" — tool dello stesso cluster (escluso il corrente)
   - "Potrebbe servirti anche" — tool da `relatedSlugs` cross-cluster (se presenti)
3. Se non ci sono cross-cluster, mostra solo la prima sezione

**Layout**:

```
┌─────────────────────────────────────────────────────────┐
│  Strumenti Correlati                                    │
│                                                         │
│  NELLO STESSO AMBITO                                    │
│  ┌──────────────────┐  ┌──────────────────┐             │

│  │    Legali        │  │    Moratori      │             │
│  │ Calcola gli...   │  │ Calcola gli...   │             │
│  └──────────────────┘  └──────────────────┘             │
│                                                         │
│  POTREBBE SERVIRTI ANCHE                                │
│  ┌──────────────────┐  ┌──────────────────┐             │
│  │ 💰 Danno         │  │ 📝 Fattura       │             │
│  │    Biologico     │  │    Avvocato      │             │

```

- Card compatte: icona + titolo + una riga di descrizione + link
- Su mobile: stack verticale
- Posizionamento: sotto il contenuto editoriale, sopra il footer, su ogni pagina tool
- Sfondo leggermente diverso (es. `bg-muted/30`) per distinguerlo dal contenuto principale

### 2.4 Breadcrumb con contesto cluster

Aggiungere un breadcrumb visivo e strutturato (JSON-LD) a ogni pagina tool.

**Visivo** (sotto la navbar, sopra l'hero):

```
Home  ›  Interessi e Rivalutazione  ›  Calcolo Interessi Legali
```

- "Home" linka a `/`
- "Interessi e Rivalutazione" linka alla hub page del cluster (quando esiste). Fino ad allora, linka alla homepage con un anchor `#interessi` che scrolla alla sezione del cluster.
- "Calcolo Interessi Legali" è testo non cliccabile (pagina corrente)

**JSON-LD** (già supportato in `src/lib/schema.tsx`):

```json
{
  "@type": "BreadcrumbList",
  "itemListElement": [
    { "position": 1, "name": "Home", "item": "https://www.avvocatotools.it/" },
    { "position": 2, "name": "Interessi e Rivalutazione", "item": "https://www.avvocatotools.it/#interessi" },
    { "position": 3, "name": "Calcolo Interessi Legali", "item": "https://www.avvocatotools.it/calcolo-interessi-legali" }
  ]
}
```

Nota: quando le hub pages esisteranno (Fase 3), il breadcrumb level 2 punterà direttamente alla hub page invece che all'anchor della homepage.

**Componente**: `src/components/shared/Breadcrumb.tsx`
- Props: `currentSlug: string` (ricava cluster e tool dal registry)
- Renderizza sia l'HTML visivo che il JSON-LD

## Fase 3: Hub/Pillar Pages

### 3.1 Cosa sono

Pagine editoriali ricche (2000-3000 parole) che fungono da "porta d'ingresso" per un intero cluster. Non sono calcolatori — sono guide complete che:
- Spiegano il macro-argomento
- Linkano a ogni tool del cluster con contesto ("usa questo calcolatore per...")
- Targettano keyword informazionali più ampie
- Includono FAQ, riferimenti normativi, casi d'uso

### 3.2 URL e routing

Ogni hub page vive alla root:

| Cluster | URL Hub |
|---|---|
| Compensi e Fatturazione | `/compensi-forensi` |
| Termini e Scadenze | `/termini-processuali` |
| Interessi e Rivalutazione | `/interessi-e-rivalutazione` |
| Danno e Risarcimento | `/danno-e-risarcimento` |
| Fiscale e Tributario | `/fiscale-e-tributario` |

Ogni hub è una `page.tsx` nella propria cartella sotto `src/app/`.

### 3.3 Struttura tipo di una hub page

```
[Breadcrumb: Home › Interessi e Rivalutazione]

[H1] Calcolo Interessi e Rivalutazione Monetaria: Guida Completa 2026

[Intro 200-300 parole]
Panoramica dell'argomento: quando servono gli interessi legali, moratori,
la rivalutazione. Chi li usa, perché, quadro normativo di riferimento.

[Sezione: I nostri strumenti]
Grid di card che linkano ai 3 tool del cluster, ognuna con:
- Titolo e descrizione
- Caso d'uso tipico ("Usa questo per calcolare...")
- Badge aggiornamento
- CTA "Vai al calcolatore →"

[Sezione: Interessi Legali — Cosa sono e come funzionano]
500-700 parole. Contenuto editoriale. Link contestuale al tool.

[Sezione: Interessi Moratori — D.Lgs. 231/2002]
500-700 parole. Contenuto editoriale. Link contestuale al tool.

[Sezione: Rivalutazione Monetaria ISTAT]
500-700 parole. Contenuto editoriale. Link contestuale al tool.

[Sezione: Differenze tra interessi legali, moratori e rivalutazione]
Tabella comparativa. Contenuto ad alto valore SEO.

[FAQ — 10-15 domande]

[RelatedTools — cross-cluster: Danno Biologico, Compenso Avvocato]
```

### 3.4 SEO delle hub pages

- H1 con keyword informazionale ampia (es. "calcolo interessi legali 2026 guida")
- Meta description 140-160 char
- JSON-LD: BreadcrumbList + Article (con immagine)
- NO WebApplication (non è un calcolatore)
- Canonical: URL della hub page
- Internal links bidirezionali: hub → tool E tool → hub (via breadcrumb + link nel testo)

### 3.5 Priorità di creazione

1. **Interessi e Rivalutazione** — 3 tool, cluster più denso, keyword informazionali ad alto volume
2. **Compensi e Fatturazione** — cluster core del sito, keyword "compenso avvocato" da 33K SV
3. **Termini e Scadenze** — "termini processuali" da 60K SV, altissimo traffico quotidiano
4. **Danno e Risarcimento** — 2 tool, buon volume
5. **Fiscale e Tributario** — 1 solo tool per ora, ha senso creare la hub quando il cluster cresce

### 3.6 Aggiornamenti post-hub

Quando una hub page è live:
- Il breadcrumb level 2 di ogni tool del cluster punta alla hub page (non più all'anchor homepage)
- La navbar: il titolo del cluster dropdown diventa cliccabile e linka alla hub
- La homepage: aggiungere "Scopri di più →" link sotto ogni sezione cluster

## Fase 4: URL Migration (futura, NON ora)

### 4.1 Quando considerarla

Valutare la migrazione URL quando si verificano TUTTE queste condizioni:
- 25+ tool pubblicati
- DA 15+ con backlink consolidati
- Tassonomia cluster stabile da almeno 6 mesi
- Dati GSC sufficienti per misurare l'impatto

### 4.2 Come eseguirla (appunti per il futuro)

- Migrare un cluster alla volta, non tutti insieme
- 301 redirect permanenti nel `next.config.ts` (sezione `redirects`)
- Aggiornare canonical, sitemap, JSON-LD, OG tags
- Monitorare GSC per 4-6 settimane dopo ogni batch
- Aggiornare i backlink esterni dove possibile (outreach)

### 4.3 Struttura URL target (ipotetica)

```
/strumenti/compensi/calcolo-compenso
/strumenti/compensi/fattura-avvocato
/strumenti/termini/calcolo-termini-processuali
/strumenti/termini/calcolo-termini-impugnazione
...
```

Questa sezione è solo un promemoria. Non implementare fino a quando le condizioni del punto 4.1 non sono soddisfatte.

## Mappa delle relazioni cross-cluster (relatedSlugs)

Queste relazioni collegano tool di cluster diversi dove esiste un flusso logico per l'utente:

| Tool | Cross-cluster suggeriti | Motivo |
|---|---|---|
| `calcolo-compenso` | `calcolo-fattura-avvocato` | Dopo il compenso l'avvocato emette fattura |
| `calcolo-fattura-avvocato` | `calcolo-compenso` | Flusso inverso |
| `calcolo-danno-biologico` | `calcolo-interessi-legali`, `calcolo-rivalutazione-monetaria` | Il risarcimento va rivalutato e produce interessi |
| `calcolo-danno-non-patrimoniale` | `calcolo-interessi-legali`, `calcolo-danno-biologico` | Stessa logica risarcitoria |
| `calcolo-interessi-legali` | `calcolo-rivalutazione-monetaria` | Spesso calcolati insieme |
| `calcolo-interessi-moratori` | `calcolo-interessi-legali` | L'utente potrebbe confondere i due tipi |
| `calcolo-rivalutazione-monetaria` | `calcolo-interessi-legali`, `calcolo-danno-biologico` | Rivalutazione usata nei risarcimenti |
| `calcolo-termini-processuali` | `calcolo-termini-impugnazione` | Flusso naturale: termine → impugnazione |
| `calcolo-termini-impugnazione` | `calcolo-termini-processuali` | Flusso inverso |
| `contributo-unificato` | `calcolo-compenso` | L'avvocato calcola CU e compenso per lo stesso caso |

## Ordine di implementazione

```
Step 1 — toolRegistry.ts
         Aggiungere CLUSTERS, campo cluster a ToolEntry, relatedSlugs, helper functions.
         Nessun impatto visivo, solo dati.

Step 2 — RelatedTools component
         Creare il componente e aggiungerlo a ogni pagina tool.
         Primo impatto UX: gli utenti iniziano a navigare tra tool.

Step 3 — Homepage raggruppata
         Riorganizzare la griglia in sezioni per cluster.
         Impatto UX sulla discovery dei tool.

Step 4 — Navbar raggruppata
         Dropdown per cluster su desktop, accordion su mobile.
         Navigazione scalabile.

Step 5 — Breadcrumb component
         Visivo + JSON-LD su ogni pagina tool.
         Impatto SEO: Google vede la gerarchia.

Step 6 — Hub pages (una alla volta, ordine di priorità dalla sezione 3.5)
         Contenuto editoriale. Aggiornare breadcrumb e navbar per puntare alla hub.
         Impatto SEO: topical authority.
```

Step 1-5 sono la Fase 1-2 e possono essere completati in sequenza rapida.
Step 6 è la Fase 3 e richiede content creation per ogni hub page.
