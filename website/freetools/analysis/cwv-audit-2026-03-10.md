## 2. Bottleneck Analysis

### 2.1 LCP: 5.5s on Both Pages (CRITICAL)

**LCP element on `/contributo-unificato`**: The hero `<div>` with CSS `background-image` pointing to the CloudFront hero banner image.

**Root causes identified by Lighthouse**:
- `fetchpriority=high` is **not applied** to the LCP resource
- The hero image is loaded via CSS `background-image: url(...)` in `HeroSection.tsx`, which means it **cannot be discovered by the HTML preload scanner** -- the browser must first download and parse the CSS, then initiate the image request
- On the landing page, the LCP element is text (the `<h1>`), but LCP is still 5.5s due to **JavaScript bundle blocking render**

**LCP Subpart breakdown (contributo page)**:
- TTFB: ~1,010ms (server too slow in dev mode for this heavy page -- 136 KiB HTML document)
- Resource Load Delay: High (CSS `background-image` not preloaded)
- Resource Load Time: Moderate (hero-banner.webp = 66.4 KiB, already WebP)
- Element Render Delay: Elevated (JS blocking rendering)

### 2.2 TTFB: 1,010ms on Contributo Page (POOR)

The contributo-unificato page HTML document is **136.3 KiB** (vs 12.9 KiB for landing). In dev mode, server-side rendering of this heavy page with many tables, data imports, and components takes over 1 second. In production with static generation (`next build`), this should drop to near-zero since the page has no dynamic data.

**However**: The production build currently fails due to a TypeScript error in `src/lib/schema.ts` line 158 (JSX in a `.ts` file -- needs to be `.tsx`).

### 2.7 Render-Blocking Resources

Landing page: **270ms** wasted on render-blocking requests.
Contributo page: **50ms** wasted.

In dev mode, Turbopack chunks are not optimally split. Production build should address most of this.

### 2.8 Caching

**262 KiB** of resources served without efficient cache lifetimes. This is expected on localhost dev server. In production, Next.js automatically adds immutable cache headers to `/_next/static/` assets.

### 2.9 Production Build Blocker

The production build fails:
```
./src/lib/schema.ts:158 - Expected '>', got 'type'
```
The file `schema.ts` contains JSX (the `JsonLd` component returns a `<script>` element) but has a `.ts` extension instead of `.tsx`. This blocks `next build` entirely, meaning the site **cannot be deployed to production**.

## 3. Prioritized Recommendations

### P0 -- CRITICAL (must fix before production)

#### 3.1 Fix `schema.ts` build error
Rename `src/lib/schema.ts` to `src/lib/schema.tsx` to fix the JSX compilation error. This is blocking any production build/deployment.

**Expected impact**: Unblocks production build, which enables static generation, code minification, tree-shaking, and all the optimizations that will dramatically improve every metric.

#### 3.2 Fix LCP hero image loading strategy
The hero image in `HeroSection.tsx` uses CSS `background-image`, which is invisible to the browser's preload scanner. Change to a proper `<img>` element (or Next.js `<Image>`) and add `fetchpriority="high"`.

**Current** (`HeroSection.tsx` line 13-15):
```tsx
<div
  className="absolute inset-0 bg-cover bg-center"
  style={{ backgroundImage: `url(${image})` }}
/>
```

**Recommended**: Replace with an `<img>` tag using `object-cover` positioning, or use Next.js `<Image>` with `priority` prop. Also add a `<link rel="preload" as="image">` in the page head for the hero image.

**Expected impact**: LCP improvement of 1-2 seconds.

### P1 -- HIGH PRIORITY

#### 3.3 Use Next.js `<Image>` for all external images
The three section images (`CIVILE_IMG`, `TRIBUTARIO_IMG`, `AMMINISTRATIVO_IMG`) use raw `<img>` tags without `width`/`height` attributes and without lazy loading. Replace with Next.js `<Image>` component:
- Automatic responsive `srcset` generation
- Automatic `width`/`height` for CLS prevention
- `loading="lazy"` by default for below-fold images
- Automatic format negotiation (AVIF when supported)

The CloudFront domain is already configured in `next.config.ts` `remotePatterns`.

**Expected impact**: ~165 KiB savings from AVIF conversion; proper lazy loading.

#### 3.4 Add `<link rel="preload">` for hero image
In the `/contributo-unificato` page (or in `layout.tsx` via `metadata`), add:
```tsx
export const metadata = {
  other: {
    'link': [{ rel: 'preload', as: 'image', href: HERO_IMG }]
  }
}
```
Or use Next.js `<Link>` in head.

**Expected impact**: Eliminate LCP resource load delay.

### P2 -- MEDIUM PRIORITY

#### 3.5 Consider reducing font count
Three fonts totaling 85.9 KiB. JetBrains Mono (39.8 KiB) is used sparingly. Options:
- Use `font-display: optional` for JetBrains Mono (falls back to system monospace if not cached)
- Load JetBrains Mono only on pages that need it (contributo-unificato)
- Subset to only needed characters (digits, euro sign, common Latin)

**Expected impact**: 20-40 KiB reduction in critical path.

#### 3.6 Lazy-load below-fold content sections
The contributo-unificato page is very content-heavy (136 KiB HTML). Consider:
- Using `React.lazy()` / `next/dynamic` for the FAQ component (interactive, below fold)
- Using `loading="lazy"` on below-fold images (already suggested above)

**Expected impact**: Reduced main-thread work, faster TTI.

#### 3.7 Consider static generation for contributo-unificato
The page has no dynamic data -- all fee tables are static. With a working production build, Next.js should statically generate this page at build time, serving it as a pre-rendered HTML file with near-zero TTFB. Verify this is happening after fixing the build error.

**Expected impact**: TTFB from 1,010ms to <50ms.

### P3 -- LOW PRIORITY (polish)

#### 3.8 Preconnect to CloudFront CDN
Add `<link rel="preconnect" href="https://d2xsxph8kpxj0f.cloudfront.net">` in layout or page head to save DNS+TLS time for image requests.

#### 3.9 Consider code-splitting Radix UI
The project includes `@radix-ui/react-accordion`, `@radix-ui/react-label`, `@radix-ui/react-select`, `@radix-ui/react-separator`, `@radix-ui/react-tabs` in dependencies, but the actual UI uses custom-built components. If these are unused, remove them to reduce bundle size.

#### 3.10 Add AVIF variants for section images
The CloudFront images are already WebP. Provide AVIF variants for Chromium browsers (~30% smaller than WebP). This can be done via CloudFront image transformation or manual upload.
